# WEB-18 · 断线重连状态恢复（2026-07-14）

分支 `web-18-restore`。

## 问题

免费旋转的次数、累计倍数、freeSpinBet、保底进度**本来就在服务端持久化**，刷新页面并不会丢局。丢的是**演出上下文**：刷新后盘面是随机 demo 假盘，配着一句"免费旋转还剩 7 次"，玩家无从判断上一局发生了什么，也不知道该点哪儿把剩下的次数打完——「开局」按钮不呼吸、文案还是"开局"，看着像一局新的。

## 契约变更（先行）

CT-2 新增 `GET /api/last-spin` → `{ spin: SpinResult | null }`，返回该玩家最后一局的完整 SpinResult。

为什么不塞进 `POST /api/session` 的 PlayerState：一是 PlayerState 是每次 spin 都要回传的热结构，塞一整个 SpinResult 会让每次 spin 的响应翻倍；二是独立端点后面 WEB-14（赢奖历史）能直接复用这条路。

契约里写清了它**只是展示恢复，不产生任何判定**：免费旋转要继续，仍须玩家点「继续」再发一次 `POST /api/spin`。

## 实施

| 层 | 改动 |
| --- | --- |
| server | `GET /api/last-spin`：按 player_id 取最新一条 spins，返回 `result_json`。未鉴权 401，没转过 → `{ spin: null }` |
| web | `restoreSession()`：开局把盘面摆成上一局最后一次 cascade 的 `gridAfter`（真实终盘，不是 demo 盘）；还欠免费旋转时弹横幅「还剩 N 次 · 累计 ×M · 点「继续」」 |
| web | 「开局」按钮在免费旋转期间改口播为「继续」；呼吸动画不再要求余额够（免费旋转不扣钱，余额再低也该招手） |
| web | `Board.currentGrid()`：只读取当前盘面，供 e2e 断言 |

## 验证

TDD 两层都**先看它失败**：

- 服务端：`apps/server/test/last-spin.test.ts` 6 项，先跑，6 红（路由不存在），再实现。
- 前端：写完 `scripts/e2e-restore.mjs` 后，把前端改动 stash 掉，在旧代码上跑 → 盘面/横幅/按钮文案/呼吸 4 项红，确认这个 e2e 真的抓得到这个 bug，再把改动恢复。

最终 `npm run e2e:restore` 12 项全过（含"恢复的盘面与服务端记录的终盘逐格一致"、"点继续后免费旋转真的接着打 10 → 9"）。七套 e2e、engine 43 + server 67 单测全绿。

### 踩到的坑

1. **像素比对判等盘面 = 假失败**。第一版 e2e 用 canvas 截图的 base64 判断"刷新前后盘面是否一致"，一直红。把两张图存下来肉眼看——**完全一样**。牌面有微光动画，逐帧像素都在变，截图还要过 CSS 缩放合成。改成读 `Board.currentGrid()` 做逐格逻辑比对，并额外与服务端 `/api/last-spin` 的终盘对账。
2. **改了服务端路由但 dev 没接管端口**。8788 上跑的是更早起的旧进程，后起的 dev 因端口占用没接管，于是新路由一直 404，我差点去查代码。教训：改路由后确认端口上的进程是不是新的（`ss -lptn`）。
3. 中间有一次整跑挂在 "TypeError: Failed to fetch"，重跑即恢复——偶发抖动，不是产品问题；如实记下，没有当成 bug 去改。

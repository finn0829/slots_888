# ENG-10 · 四档 RTP 预设重校 + 分析器偏差修正（2026-07-14）

## 起因

规则页对玩家公示"RTP 约 95.6%"，而配置自称 96.5 档，ENG-6b 的报告又说实测 97.18%——**三个数互相打架**。四档预设（92 / 94.5 / 96.5 / 97.5）的 payoutScale 还是最早直测标定的，误差 ±1~2pp。公示数字和真实返奖率对不上，性质上和"隐藏动态 RTP"是同一类问题，属于概率诚实原则的红线范围。

## 过程中发现的真问题：分析器**有偏**，不只是有噪声

原打算直接用 ENG-6b 的 `analyze.ts` 重标一遍。但标定结果**自相矛盾**：

```
rtp92   目标 92.00% → scale 0.9452 → 复测 93.82%（+1.82pp）
rtp945  目标 94.50% → scale 0.9709 → 复测 92.89%（−1.61pp）   ← 更高的 scale，更低的 RTP？
```

更低的赔付倍数测出更高的 RTP，物理上不可能。顺着这个矛盾查下去，挖出三层问题：

### ① 触发率是采样估的，噪声压过了信号

免费旋转触发率 1/153，120 万局也只见到约 7800 次，相对误差 1.1%；作用在占 RTP 六成的免费旋转通道上就是 ±0.7pp，比档位间距还大。

**修**：`trigger.ts` 的二项分布解析解（ENG-6b 时已经写了，但 `analyze()` 没用上）。骰子率同理——骰子只在初盘面出现，`E[骰子/局] = 30 × p` 也是解析值。这两项噪声直接清零。

### ② 误差条漏了两项，把误差系统性低估

- **保底段价值**的噪声：保底占 RTP 三成，它的价值是采样量，但完全没进误差条；
- **基础局 RTP** 的噪声：基础局同样有长尾大奖（可到 5000×），也没算。

结果就是"锚点 ±0.10pp，换个 seed 却差 0.6pp（7σ）"这种自相矛盾。**一个盖不住实际离散度的误差条，比没有误差条更危险**——它让人以为自己在精调，其实在追噪声。

**修**：三个采样量各自算方差再合成。现在六次独立估计的实际离散度 0.19pp，报告的 stderr 0.29pp，盖得住。

### ③ 最严重的：保底通道**算法本身是错的**（偏差 1.7pp）

原公式：

```
pityValue = E[单段免费旋转价值] × (10 / avgAward)      // 按次数线性折算
```

它假设"一段免费旋转的价值随次数线性缩放"。**它离线性差得很远**——累计倍数在整段里滚雪球（还会再触发），段越长越值钱。实测（各 15 万段）：

```
10 次的段 →  42.6×
20 次的段 → 197.7×     线性应为 85.1×，实际高出 132%
```

于是保底被系统性高估，基准档 RTP 被报成 **97.4%，真值 95.9%**。

**为什么没被早发现**：直接蒙特卡洛救不了——3M 局直测的标准误差就有 ±1.1pp，**分辨不出 1.7pp 的偏差**。所以我加的守卫不是"跟直测对一下"（那道闸太松，两个版本都能过），而是直接**证伪那个线性假设**：`analyze-unbiased.test.ts` 断言 `v20 > 2.1 × v10`。

**修**：保底段按 10 次直接模拟，不做任何外推。

### 顺带纠正一个方法论错误

ENG-6b 报告写着"各档用相同 seed 前缀，误差在差值中抵消"。**不成立**：`spin.ts` 在赢奖触顶 5000× 时提前终止连锁，是否触顶取决于 payoutScale ⇒ 改 scale 会改变 RNG 消耗，后续盘面整条路径发散。同一个 seed 在不同配置下不是同一条路径，配对比较无效。

## 重校结果

样本：250 万基础局 + 250 万免费旋转段/档，估计噪声 **±0.15pp**。方法：锚点线性外推 → 实测校正一轮。

| 档位 | 目标 | payoutScale 系数 | 实测 RTP | 偏差 |
|---|---|---|---|---|
| rtp92 | 92.0% | 0.9561 | **92.24%** ± 0.15 | +0.24pp |
| rtp945 | 94.5% | 0.9793 | **94.73%** ± 0.15 | +0.23pp |
| rtp965（默认） | 96.5% | 1.0 | **96.26%** ± 0.15 | −0.24pp |
| rtp975 | 97.5% | 1.0142 | **97.10%** ± 0.15 | −0.40pp |

全部落在验收要求的 ±0.5pp 内。基准配置 `payoutScale` 0.42 → **0.4224**（默认档现在真的就是 96.5 档——此前 `defaultPreset()` 自称 rtp965，实际是未缩放的锚点）。

## 公示 RTP 不再写死

规则页原本硬编码"约 95.6%"。现在 `/api/config` 下发 `rtp = estimated_rtp ?? nominalRtp`：

- 预设配置 → 用标定值（`nominalRtp`，即上表实测值）
- 管理员改过权重并跑过模拟器的版本 → 用模拟器估算值（此时标定值已失效）

后台把生效档位切到 92，玩家规则页的公示值立刻变成 92.2%（`e2e:rtp` 实测验证）。**写死的数字改一次配置就成了谎言**——Ante 的触发率此前已经这么处理，这次把 RTP 也收编了。

顺带发现服务端首次建库时往 `estimated_rtp` 塞了一个旧直测值 0.9558，它会盖住标定值。已改为留空。

> ⚠️ **迁移提醒**：已有的 `slots.db` 里存的是旧配置（payoutScale 0.42、estimated_rtp 0.9558），不会自动更新。要用上新标定，删库重建，或在后台从预设新建草稿并发布。

## 对 ENG-6b 结论的影响（重要）

Ante 的 RTP 优势是**偏差造出来的假象**：

| | ENG-6b（有偏） | 修正后 |
|---|---|---|
| 基础档 RTP | 97.18% | 95.85% ± 0.15 |
| Ante 档 RTP | 98.08% | 95.83% ± 0.26 |
| Ante − 基础 | **+0.90pp** | **+0.14pp（0.4σ，与 0 无异）** |

诚实的表述是：**Ante 的 RTP 与基础档基本持平，加注买到的是"免费旋转触发快 1.64 倍"，不是更高的返奖率。**

所幸游戏内的 Ante 文案自始至终只承诺"免费旋转更快"，从没向玩家承诺过更高 RTP——**没有对玩家撒过谎**。`anteScatterFactor = 1.16` 保持不变：多付 25% 换 1.64× 的触发速度、RTP 不吃亏，这个交易本身是公道的。ENG-6b 报告已加勘误。

## 验证

- 单测：engine 58（新增 `analyze-precision` 5 项、`analyze-unbiased` 3 项、`preset-rtp` 7 项）、server 70（新增 `public-rtp` 3 项）
- e2e：八套全绿，新增 `e2e:rtp`——后台切档 → 玩家规则页公示值跟着变
- 估计器自检：六次独立估计离散度 0.19pp vs 报告 stderr 0.29pp，误差条盖得住

## 遗留

- 免费旋转整段总赢仍可超过 5000× 单局封顶（CT-1 v0.2 已记，是否改为整段封顶待定）
- `analyze()` 的 stderr 略保守（0.29 vs 实际 0.19），因为三项误差按独立相加，实际略有负相关
